Reasons to Use Trongate

Reason #3 : No Third-Party Dependencies

Whilst the usage of third-party code can sometimes be a good thing, it should never be an integral part of any core web development framework. Any framework that depends upon third-party code is subjected to:

  • Somebody else's rewrite schedule.
  • Somebody else's security challenges.
  • Somebody else's vision.

Why Framework Makers Cosy Up To Third-Party Libraries

By declaring a friendly association with a third-party asset (such as JavaScript library), a framework maker gets the benefit of being able save time by using somebody else's work. Perhaps even more importantly, however, is the reality that these types of associations often result in astonishingly effective framework promotion.

A declaration that "Our framework chooses Library 'x'" would almost certainly result in the makers of 'Library x' telling their following about your framework. This will likely be picked up and reported by other publishers of web development content. The result is a sort of cosy cross-asset promotion with one effectively side saying:

  • "Hey, all you users of Library X - you might want to check out our PHP framework because we're using what you have!"
And the other side saying,
  • "Hey everybody, look at how popular we've become - this framework has started using our code!"

Trongate Choose The Difficult Path

Instead of cosying up to third-party providers, the makers of Trongate choose a far more difficult path. We did the unthinkable and we wrote our own libraries! This includes:

  • A compete eco-system of JavaScript components
  • A minimalist CSS framework
  • A code sharing eco-system
  • A graphical query builder

Having our own libraries means that Trongate developers can sleep comfortably at night, knowing that they're not going to wake up to the news that all their precious work has suddenly become obsolete - all because of somebody else's rewrite schedule!
